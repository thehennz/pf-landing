# SSL-сертификаты Let's Encrypt — практическое руководство

Как настраивать бесплатные SSL-сертификаты на сервере, где уже что-то работает
(VPN, сайт, панель) — без конфликтов и с автопродлением.

---

## Часть 1. Как это вообще работает (теория за 3 минуты)

**SSL-сертификат** — это файл, который подтверждает, что домен реально твой,
и позволяет шифровать трафик (https вместо http). Браузеры доверяют сертификатам
от **Let's Encrypt** — бесплатного центра сертификации.

Чтобы Let's Encrypt выдал сертификат, нужно **доказать, что домен принадлежит тебе**.
Это называется «валидация» (проверка). Есть три способа:

| Способ | Как проверяет | Что нужно | Минус |
|---|---|---|---|
| **HTTP-01** | Подключается к твоему серверу на **порт 80** и читает файл | Свободный порт 80, открытый из интернета | Порт 80 должен быть свободен |
| **TLS-ALPN-01** | Подключается на **порт 443** особым образом | Свободный порт 443 | Порт 443 должен быть свободен |
| **DNS-01** | Проверяет **TXT-запись в DNS** домена | Доступ к API DNS (токен) | Чуть сложнее настроить |

### Главный вывод

> Если на сервере уже что-то занимает порты 80 и 443 (например, VPN),
> используй **DNS-01** — он не требует ни одного порта и ни с чем не конфликтует.

Именно поэтому для сервера с Marzban/VPN правильный выбор — **DNS-валидация**.

## Часть 2. Разбор нашего случая (что пошло не так и как починили)

### Исходная ситуация
- Порт **443** занимал `xray` — движок VPN (Marzban).
- Сертификат для VPN выдавал **acme.sh** способом **HTTP-01** (через порт 80, режим «standalone»).
- Мы захотели поставить рядом сайт по HTTPS.

### Проблема
Сайт (Caddy) занял порт **80 насовсем**. А acme.sh для продления сертификата VPN
тоже хотел порт 80 (на пару секунд раз в 60 дней). Порт занят → продление бы провалилось →
через ~2 месяца сертификат VPN истёк бы → **VPN сломался бы сам по себе**.

### Как нашли
```bash
# Посмотреть, кто занимает порты 80 и 443
ss -tlnp | grep -E ':(80|443) '

# Узнать, каким способом acme.sh продлевает сертификат
grep Le_Webroot ~/.acme.sh/ДОМЕН_ecc/ДОМЕН.conf
```
Увидели `Le_Webroot='no'` — это режим **standalone (порт 80)**. Вот она, мина.

### Как починили
Переключили acme.sh с порта 80 на **DNS-валидацию** (через API DuckDNS).
Теперь порт 80 ему не нужен, конфликт исчез.

## Часть 3. Пошаговая настройка сертификата для VPN (на будущее)

Допустим, у тебя сервер с VPN на порту 443 и домен DuckDNS. Вот полный путь.

### Шаг 1. Установить acme.sh (если ещё нет)

```bash
curl https://get.acme.sh | sh -s email=твоя@почта.com
```
Перелогинься в консоль (или `source ~/.bashrc`), чтобы команда `acme.sh` заработала.

> Проверить, установлен ли уже: `ls ~/.acme.sh/`

### Шаг 2. Узнать токен DuckDNS

Зайди на [duckdns.org](https://www.duckdns.org), вверху страницы строка `token=...`.
Это и есть твой токен (один на все домены аккаунта).

### Шаг 3. Выпустить сертификат через DNS

```bash
export DuckDNS_Token="ТВОЙ_ТОКЕН"
acme.sh --issue --dns dns_duckdns -d твойдомен.duckdns.org --ecc
```

Что происходит: acme.sh добавляет TXT-запись через API DuckDNS, Let's Encrypt
её проверяет, выдаёт сертификат. **Порты не нужны.** Токен сохранится — при будущих
продлениях вводить заново не надо.

> `--ecc` — современный компактный тип ключа (рекомендуется).
> Если переключаешь СТАРЫЙ сертификат с порта 80 на DNS — добавь `--force`.

### Шаг 4. Положить сертификат туда, где его читает приложение

У каждого приложения своё место для сертификата. Для Marzban это `/var/lib/marzban/certs/`.
Команда `--install-cert` копирует сертификат туда и настраивает **авто-перезапуск**
приложения после каждого продления:

```bash
acme.sh --install-cert -d твойдомен.duckdns.org --ecc \
  --key-file /var/lib/marzban/certs/твойдомен.duckdns.org.cer.key \
  --fullchain-file /var/lib/marzban/certs/твойдомен.duckdns.org.cer \
  --reloadcmd "docker restart ИМЯ_КОНТЕЙНЕРА"
```

- `--key-file` / `--fullchain-file` — куда положить ключ и сертификат (смотри в доке своего приложения).
- `--reloadcmd` — команда перезапуска. Узнать имя контейнера: `docker ps`.
- ⚠️ При запуске этой команды приложение разок перезапустится (проверка, что хук рабочий).

### Шаг 5. Проверить, что всё на месте

```bash
# Метод должен быть dns_*, а не 'no'
grep Le_Webroot ~/.acme.sh/твойдомен.duckdns.org_ecc/твойдомен.duckdns.org.conf
# Ожидаем: Le_Webroot='dns_duckdns'

# Когда следующее продление
grep Le_NextRenewTimeStr ~/.acme.sh/твойдомен.duckdns.org_ecc/твойдомен.duckdns.org.conf
```

Готово. Дальше acme.sh всё делает сам — продлевает раз в ~60 дней через DNS
и перезапускает приложение.

## Часть 4. Как работает автопродление

acme.sh при установке добавляет задание в cron (планировщик):
```bash
crontab -l | grep acme
# 49 4 * * * "/root/.acme.sh"/acme.sh --cron --home "/root/.acme.sh" > /dev/null
```
Каждый день в 4:49 acme.sh проверяет все сертификаты. Если какому-то пора (осталось < 30 дней) —
продлевает его тем способом, что записан в конфиге (у нас — DNS), и запускает `reloadcmd`.

**От тебя не требуется ничего.** Сертификаты Let's Encrypt живут 90 дней,
acme.sh обновляет их каждые 60 — с запасом.

## Часть 5. Полезные команды и диагностика

```bash
# Список всех сертификатов acme.sh и сроки
acme.sh --list

# Какой сертификат реально отдаёт сервер на порту (проверка «снаружи»)
echo | openssl s_client -connect твойдомен.duckdns.org:443 2>/dev/null \
  | openssl x509 -noout -issuer -dates

# Кто занимает порты 80 и 443
ss -tlnp | grep -E ':(80|443) '

# Принудительно продлить сертификат прямо сейчас (для теста)
acme.sh --renew -d твойдомен.duckdns.org --ecc --force

# Посмотреть конфиг домена целиком
cat ~/.acme.sh/твойдомен.duckdns.org_ecc/твойдомен.duckdns.org.conf
```

> Примечание: в новых версиях acme.sh поле `Le_ReloadCmd` хранится в виде
> `__ACME_BASE64__START_...__END_` — это нормально, команда просто закодирована
> в base64 для безопасности. Раскодировать: `echo "СТРОКА" | base64 -d`.

## Часть 6. Частые проблемы

| Симптом | Причина | Решение |
|---|---|---|
| `Bind for 0.0.0.0:80 failed: port is already allocated` | Порт 80 занят другим сервисом | Освободить порт ИЛИ перейти на DNS-валидацию |
| Продление падает с ошибкой port 80 | acme.sh в режиме standalone, а порт занят | Переключить на DNS: `--issue --dns dns_duckdns ... --force` |
| Сертификат продлился, но приложение отдаёт старый | Нет `reloadcmd` — приложение не перезапустилось | Добавить `--install-cert ... --reloadcmd "..."` |
| DNS-проверка не проходит | Неверный токен DuckDNS | Проверить токен на duckdns.org |
| `too many certificates already issued` | Лимит Let's Encrypt (5 одинаковых/неделю) | Подождать неделю, не дёргать `--force` без нужды |

## Шпаргалка: перевести существующий сертификат с порта 80 на DNS

Если сертификат уже есть, но продлевается через порт 80 (`Le_Webroot='no'`),
и нужно убрать зависимость от порта:

```bash
export DuckDNS_Token="ТВОЙ_ТОКЕН"
acme.sh --issue --dns dns_duckdns -d твойдомен.duckdns.org --ecc --force
```
Одна команда — и метод сменится на DNS. Деплой-настройки (куда копировать, что
перезапускать) при этом сохраняются.
